Brifen ba om en plan for v4.0. Tidlige iterasjoner viste at agenten var klar til å programmere før vi hadde snakket om UI og kode-arkitektur. Etter samtale 2026-02-XX bestemte vi:

## 1. Mental modell (én setning)

> **Ko|Do Vault får en valgfri pakke-modul som lar deg pakke filer inn i en kryptert `.kodoenc`-fil — eller pakke ut en `.kodoenc` du har mottatt. Modulen er den samme om du er innlogget eller ikke; bare "pakk inn" krever login.**

## 3. Toggle-modell (`default.json`)

```jsonc
{
  "features": {
    "packages": {
      "showOnLogin": true,         // Anna-knapp "Pakk ut" på login-siden
      "showInApp": true,           // Lars-knapp 📦 i header + MobileBottomBar
      "maxFileSizeMB": 50,         // Standard: 50 MB (iPhone-trygt). Tenant-eier kan heve. Ingen clamp.
      "allowedMimeTypes": null     // null = alle. Sett til array for restriksjon pr tenant
    }
  }
}
```

**Hvorfor TO toggles:**
- En tenant kan ville la mottakere (Anna) bruke utpakking uten å la sine egne vault-brukere lage nye pakker. F.eks. en kunde som kun mottar pakker, ikke sender.
- En tenant kan ville la sine vault-brukere lage pakker uten å eksponere "Pakk ut"-knappen på login. F.eks. et lukket selskap der ingen utenforstående skal sende noe inn.
- Begge `false` = ingen pakke-UI noen sted.
- Begge `true` = full v4.0-funksjonalitet (standard for Mike/default.json).

**Tenant-konsekvenser:**
- `default.json` (Mike): begge `true`, **50 MB**
- `default-lk.json` (Lisbeth): kan settes individuelt
- Fremtidige tenants kan styre per kunde

**`features.packages.enabled: false`:**
- Login-siden: ingen "Pakk ut"-knapp
- I appen: ingen 📦-ikon i header
- `/api/` påvirkes ikke (vi har ingen API for pakker — alt er klient-side)

## 4. Filformat — `.kodoenc` envelope

### 4.1 Strukturen (binær)

`.kodoenc` er **én enkelt binærfil** med følgende struktur:

```
┌─────────────────────────────────────────────────────────────┐
│ MAGIC HEADER (8 bytes)        "KODOENC\0"                   │
├─────────────────────────────────────────────────────────────┤
│ VERSION (2 bytes, uint16-LE)  0x0001  (v4.0)                │
├─────────────────────────────────────────────────────────────┤
│ HEADER-LENGTH (4 bytes, uint32-LE)  → N bytes header-JSON   │
├─────────────────────────────────────────────────────────────┤
│ HEADER-JSON (N bytes, UTF-8) — KLARTEKST (minimum strukturell): │
│ {                                                            │
│   "kind": "kodo-package",                                   │
│   "version": 1,                                             │
│   "kdf": {                                                  │
│     "algorithm": "PBKDF2-SHA256",                           │
│     "iterations": 600000,                                   │
│     "saltB64": "..."                                        │
│   },                                                        │
│   "cipher": {                                               │
│     "algorithm": "AES-256-GCM",                             │
│     "ivB64": "...",                                         │
│     "tagBits": 128                                          │
│   }                                                          │
│ }                                                            │
│ MERK: createdAt, appVersion, fileCount, container-type      │
│ ligger inne i kryptert payload (_metadata.json i ZIP).      │
│ Klartekst-header inneholder KUN det mottaker MÅ ha for å    │
│ derive nøkkelen + decrypt. Se D-025 (revidert v4.0-spec).   │
├─────────────────────────────────────────────────────────────┤
│ CIPHERTEXT (resten av fila)                                 │
│ AES-256-GCM-kryptert ZIP STORE-arkiv som inneholder:        │
│   ├── _metadata.json   (createdAt, appVersion, fileCount)   │
│   ├── fil-1.pdf                                             │
│   ├── fil-2.pdf                                             │
│   └── Hoppeslott/Underlag/skanning.pdf  ← bevart mappe-struktur │
│                                                              │
│ ⚠ AES-GCM auth-tag (16 bytes = 128 bits) er IMPLISITT       │
│   appendet til ciphertext av Web Crypto API.                │
│   Format: [encrypted-zip-bytes] || [16-byte-tag]            │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 Hvorfor minimum klartekst-header?

- Header inneholder KUN det mottakers nettleser MÅ ha for å derive nøkkelen + decrypt: KDF-parametre (algorithm, iterations, salt) + cipher-parametre (algorithm, iv, tag-bits)
- `createdAt` + `appVersion` + `fileCount` flyttes inn i kryptert `_metadata.json` (se D-025) — bruks-metadata, ikke strukturell. Ingen klartekst-fingerprint av Lars sin pakke-aktivitet.
- Pakke-navnet er kun filnavnet på disken (Mike sitt valg) — Lars' eget ansvar
- Versjons-flagget gir oss bakoverkompatibilitet for fremtidige format-endringer

### 4.3 Krypterings-flyten (avsender)

```
1. Bruker velger filer + pakke-navn
2. Bygg ZIP STORE-arkiv i RAM (JSZip med compression: STORE — D-007):
   - Inkluder originale filer (bevar mappe-struktur via webkitRelativePath)
   - Inkluder _metadata.json med { createdAt, appVersion, fileCount, container }
3. Generer salt (16 bytes random)
4. Generer iv (12 bytes random)
5. Bruker taster engangs-passord (eller bruker "🎲 Generer"-knapp)
6. Derive key = PBKDF2(pwd, salt, 600k iter, SHA-256)
7. ciphertext = AES-256-GCM-encrypt(zipBytes, key, iv)
   → Web Crypto API legger 16-byte auth-tag automatisk på slutten
8. Bygg envelope: magic + version + headerLen + headerJSON + ciphertext
9. Last ned som Blob → <a download={pakke-navn + ".kodoenc"}>
```

### 4.4 Dekrypterings-flyten (mottaker)

```
1. Bruker drar fil inn → leses som ArrayBuffer
2. Verifiser magic-bytes "KODOENC\0" — hvis feil → "Ikke en gyldig Ko|Do-pakke"
3. Les version → 0x0001 → forventet (større = "oppgrader appen")
4. Les headerLen → parse headerJSON → valider KDF + cipher-felt
5. Bruker taster pwd
6. Derive key = PBKDF2(pwd, salt fra header, iterations fra header)
7. zipBytes = AES-256-GCM-decrypt(ciphertext, key, iv)
   → Web Crypto API verifiserer 16-byte auth-tag automatisk
   → Feil pwd ELLER tampered ciphertext → "Feil passord eller korrupt fil"
8. JSZip.load(zipBytes) → file-tree
9. Les _metadata.json (klient-side, kun for diagnostikk hvis bruker vil se "laget med v4.0.0")
10. Vis filer i UI (flat liste med full sti som filnavn) → preview + last ned
```

### 5.3 State-isolering

- `<UnpackModule />` har INGEN tilgang til `useVault`-hook
- `<PackModule />` har INGEN tilgang til `useVault`-hook (selv om den brukes inne i appen — pakkingen er separat fra vault-data)
- Dvs. begge modulene er fullstendig "stateless" mhp. vault og kan rendres på login-siden uten å trekke inn vault-state

## 6. UI-design (glass-arkitektur per D-022/D-023)

### 6.1 Regler vi MÅ følge (kritisk for Safari)

- ✅ Alle modaler bruker `.backdrop-blur-xl` (plukker `--kodo-card-bg` + `--kodo-blur-xl` fra `:root` per browser)
- ✅ Modaler er **aldri** direkte flex-barn av `<main>` — wrap i `<div className="w-full max-w-md">` eller bruk en portal-mounted modal
- ✅ Aldri hardkode `bg-white/10` eller egen `backdrop-filter` i nye komponenter
- ✅ Aldri `isolation: isolate` på bg-wrapper

### 6.2 Visuelle valg

- Pakker-knapp på login: under master-pwd-feltet, samme bredde, mindre prominent (sekundær aksjon)
- 📦-ikon i header: samme stil som eksisterende header-ikoner (Lucide-icons)
- Engangs-passord vises i monospace-font med tydelig pill-styling
- Drop-zones: dashed border i `--kodo-card-bg`, hover-state løfter shadow
- **Pakke-navn-felt** har under-tekst som hint: *"Bruk gjerne generisk navn (`pakke-2026-02-15`) hvis pakka skal være mest mulig anonym. Filnavnet vises i klartekst hos mottaker."*
- **Pakke-pwd-felt på mottaker-siden** har tydelig label: *"🔒 Pakke-passord (IKKE ditt vault-passord)"* — kritisk for å hindre at innloggede brukere taster sitt eget master-pwd

### 6.3 Lars-valgt engangs-passord

```
Inntastingsfelt:  Lars taster selv (sin valgte sikkerhetsnivå)
Hjelpe-knapp:     "🎲 Generer sterkt" — gjenbruker eksisterende Passord-laben
                  (lib/password-generator.ts + lib/passphrase-generator.ts)
Styrke-meter:     zxcvbn under feltet (samme som master-pwd-setup) — gir Lars
                  visuell feedback men blokkerer ikke svake valg (Lars sitt ansvar)
Min lengde:       8 tegn (hard gulv — under det er det meningsløst)
Visning:          Klartekst med 👁-toggle (Lars må se det for å lese opp i tlf)
```

## 9.1 Multi-container-design (uavhengige containere)

```
                  Lars splitter 12 filer i 3 containere
                                  │
        ┌─────────────────────────┼─────────────────────────┐
        ▼                         ▼                         ▼
   Hoppeslott-1.kodoenc      Hoppeslott-2.kodoenc      Hoppeslott-3.kodoenc
   ┌──────────────┐          ┌──────────────┐          ┌──────────────┐
   │ Magic header │          │ Magic header │          │ Magic header │
   │ Klartekst    │          │ Klartekst    │          │ Klartekst    │
   │   JSON-conf  │          │   JSON-conf  │          │   JSON-conf  │
   │   (egen salt,│          │   (egen salt,│          │   (egen salt,│
   │    egen IV)  │          │    egen IV)  │          │    egen IV)  │
   │ ───────────  │          │ ───────────  │          │ ───────────  │
   │ Ciphertext:  │          │ Ciphertext:  │          │ Ciphertext:  │
   │   fil-1.pdf  │          │   fil-5.pdf  │          │   fil-9.pdf  │
   │   fil-2.pdf  │          │   fil-6.pdf  │          │   fil-10.pdf │
   │   fil-3.pdf  │          │   fil-7.pdf  │          │   fil-11.pdf │
   │   fil-4.pdf  │          │   fil-8.pdf  │          │   fil-12.pdf │
   └──────────────┘          └──────────────┘          └──────────────┘
   Anna åpner alene          Anna åpner alene          Anna åpner alene
   → ser fil 1-4              → ser fil 5-8              → ser fil 9-12

   Samme passord brukes for alle 3 (Lars sitt valg).
   Hvert salt + iv er unik (kryptografisk hygiene — D-001).
```

**Konsekvenser av uavhengighet:**
- Hver fil er KOMPLETT i sin container (vi splitter aldri på byte-nivå, kun på fil-nivå)
- Hvis én container går tapt på e-post → Anna får fortsatt se de andre 8 filene
- Anna kan jobbe parallelt: åpne container 1 mens container 2-3 fortsatt lastes ned
- Hver container kan ha forskjellig størrelse (Lars eller appen velger beste split basert på filstørrelser)
- Mottaker-UI er identisk for 1, 2 eller 3 containere — hun drar én fil inn av gangen

**Fordeling av filer pr container:**
- Default-strategi: minste-residual-bin-packing (filer pakkes inn slik at containerne blir omtrent like store)
- Lars kan se forhåndsvisning i Iter 2 UI: *"Container 1: 4 filer (8 MB), Container 2: 4 filer (12 MB), Container 3: 4 filer (10 MB)"*
- Hvis én enkeltfil er større enn ønsket container-størrelse → den får sin egen container (vi splitter aldri på fil-nivå)

## 9.4 Zip-Slip-beskyttelse + filnavn-sanitering + mappe-struktur

**Trusselen:** En ondsinnet `.kodoenc`-fil kunne inneholde filnavn som `../../../../../etc/passwd` eller `C:\Windows\System32\evil.exe`. Hvis Anna's nettleser blindt skriver disse via File System Access API → skadelig file-overwrite.

**Hvordan v4.0 håndterer mapper:**
- ZIP-arkivet **bevarer mappe-struktur** (gratis via JSZip — Lars drar en mappe inn, browseren leverer `webkitRelativePath`)
- Mottaker-UI viser **flat liste** med full sti som filnavn (`Hoppeslott/Underlag/skanning.pdf`)
- Når Anna trykker "Last ned" eller bruker "Velg utpakkings-mappe" via FSAccess, gjenskapes mappestrukturen
- Hvert path-segment SANITERES individuelt mot zip-slip

**Mottaker-siden (`<UnpackModule />`):**

```typescript
// lib/package-zip.ts — sanitizeZipPath
function sanitizeZipPath(rawPath: string): string {
  // 1. Normaliser backslash → slash
  let s = rawPath.replace(/\\/g, "/");
  // 2. Strip ledende slashes og ../
  s = s.replace(/^(\.\.\/|\/)+/g, "");
  // 3. Split på "/", saniter hver enkelt segment
  const segments = s.split("/").map(seg => {
    // Avvis "..", ".", tomme segmenter
    if (seg === ".." || seg === "." || seg === "") return null;
    // Filtrer kontrolltegn + Windows-forbudte tegn
    let cleaned = seg.replace(/[\x00-\x1F<>:"|?*\\]/g, "_");
    // Begrens lengde pr segment
    if (cleaned.length > 100) cleaned = cleaned.slice(0, 100);
    return cleaned;
  }).filter(Boolean);
  // 4. Begrens total path-dybde
  if (segments.length > 8) segments.length = 8;
  // 5. Begrens total lengde
  let result = segments.join("/");
  if (result.length > 300) result = result.slice(0, 300);
  // 6. Tom-streng fallback
  return result || "ukjent-fil";
}
```

**Avsender-siden (`<PackModule />`):**
- Vi pakker `webkitRelativePath` (eller `file.name` hvis ikke fra mappe-drop)
- JSZip lagrer hierarkiet 1:1 inn i arkivet
- Saniterer IKKE på avsender-siden — det er mottakerens trust-grense (mottaker kan motta pakke fra fremmed kilde)

**Verifisering:** Egen test i Iter 1: `package.test.ts → "zip-slip-attempt-is-sanitized"` lager en pakke med `../../etc/passwd`-navn programmatisk og verifiserer at utpakking gir `etc/passwd` (segmenter sanitert, ikke escape ut av root).

## 9.10 MIME-type-preview-logikk

Hvilke filer får `[👁 Forhåndsvis]`-knapp avgjøres av **filendelse** (ikke content-sniffing, som er upålitelig i nettleser):

| Type | Endelser | Hvordan vises |
|---|---|---|
| PDF | `.pdf` | PDF.js inline-renderer (lazy-import) |
| Bilde | `.jpg`, `.jpeg`, `.png`, `.webp`, `.gif`, `.svg` | `<img src={objectUrl}>` |
| Tekst | `.txt`, `.md`, `.csv`, `.json`, `.log` | `<pre>{await blob.text()}</pre>` med scrollbox |
| Andre | alle andre | Kun `[↓ Last ned]`-knapp, ingen preview |

**Begrunnelse for filendelse-basert:**
- Pålitelig (vi har full kontroll på filendelsen vi lagrer i pakka)
- Anna ser tydelig hva som forhåndsvises (PDF-ikon, bilde-ikon, tekst-ikon)
- Content-sniffing kan gi falske positive (en `.exe` med PDF-magic-bytes ville vist seg som PDF)

**RAM-bevissthet:** Preview åpner én fil av gangen i RAM. Lukkes ved navigasjon. Ingen flere filer åpnes samtidig.

## 9.11 Test-strategi (Iter 1 + senere)

**`lib/__tests__/package.test.ts` (Iter 1) — minst 9 tester:**

7. ✅ **Multi-container roundtrip (3 containere)** — pakk 12 filer i 3 containere med samme pwd → utpakk hver container alene → alle får sin del korrekt
8. ✅ **Zip-Slip-forsøk saneres** — pakk en fil med navn `../../etc/passwd` → utpakk → path-segmenter sanitert, ingen escape ut av root
9. ✅ **Tom fil-liste avvises** — bygg pakke med 0 filer → kaster `Error("Må inneholde minst én fil")`
