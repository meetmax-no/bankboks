**Status:** Pre-design — beslutninger og åpne spørsmål samles her før implementasjons-ADR skrives.
**Dato:** 2026-02
**Implementasjon:** Tidligst etter v4.3 (språkdrakt).
**Forutsetninger:** v4.0 (sikker overlevering) ferdig. Pakke-format og krypto-primitiver gjenbrukes.

> Dette er et levende arbeidsdokument. Når v4.5 startes konkret, ekstraheres beslutningene herfra til formelle ADR-er (D-033, D-034, …) og dokumentet markeres som ferdig.

## 1 · Kjernekonsept — todelt arkitektur

Ko|Do har to lag med ulike krav. v4.5 introduserer Lag 2:

| Lag | Hva | Størrelse | Bruksfrekvens | Lagring |
|---|---|---|---|---|
| **1 · Hurtig-laget** | Passord, kort, ID-er | KB-MB | Mange × per dag | Upstash (Ko\|Do sin sky) |
| **2 · Dokument-laget** | PDF, Word, notater, store filer | MB-GB | Sjelden | **BYO Drive** (v4.5) |

**Lars sin mentalmodell:**
> «Jeg har én vault. Den har én master-passord. Inne i den ligger mine passord lett tilgjengelig, og mine dokumenter i en sikker mappe. Jeg vet ikke (og bryr meg ikke om) hvor dataene fysisk ligger — Ko\|Do håndterer det.»

**Drive ser:** Én binær fil (eller en samling .enc-filer i en skjult mappe). Aldri klartekst.

## 2 · Container-modell (kjerne-design)

### 2.1 Struktur

```
Drive/ (eller Drive App-data hvis vi velger drive.appdata-scope):
├── catalog.enc            ~50-200 KB   (alltid lastet — vet om ALT)
├── container-001.enc      ~5-10 MB
├── container-002.enc      ~5-10 MB
├── container-003.enc      ~5-10 MB
├── ...
└── container-N.enc        ~5-10 MB
```

### 2.2 Katalogen er hjernen

`catalog.enc` inneholder:
- Hele virtuell mappe-struktur (tre)
- Per fil: navn, størrelse, dato, mime-type, tags, container-referanse
- Eventuelt thumbnails (eller egen `thumbnails.enc`-container)
- **Lars sin tag-database** (manuelt tildelte tags per fil)
- Søke-index basert på filnavn + tags

### 2.3 Containerne er dumme

Hver container er en mini `.kodoenc` (gjenbruker v4.0-format eller ny versjons-byte 0x10):
- Samler N filer i én kryptert blob
- ZIP STORE-organisering inni (per v4.0 D-022)
- Selv-beskrivende header: liste over filnavn (for katalog-recovery)

### 2.4 Eksempel-flyt (lese)

```
Lars åpner Ko|Do  →  henter catalog.enc (~100 KB, øyeblikkelig)
                  →  rebuilder virtuell mappe-tre i UI
Lars klikker "Møter 2025 / Kontrakt-Olsen.pdf"
                  →  catalog viser: "ligger i container-007"
                  →  henter container-007.enc (~8 MB)
                  →  dekrypterer i RAM
                  →  ekstraherer kontrakt.pdf som Blob
                  →  viser i PDF-preview
                  →  container-007 caches i RAM resten av sesjonen
```

### 2.5 Eksempel-flyt (skrive en fil)

```
Lars erstatter "Kontrakt-Olsen.pdf" (samme container-007, samme størrelse)
                  →  re-krypterer container-007 (8 MB)
                  →  uploader hele container-007.enc til Drive
                  →  oppdaterer catalog.enc (lastModified-felt)
                  →  uploader catalog.enc

INGEN andre containere røres. Lars endrer aldri "hele disken sin".
```

### 2.6 Container-egenskaper Lars ALDRI ser

Containere er **implementasjons-detalj**. UI-en eksponerer aldri:
- Ord som "container", "blob", "katalog"
- Hvilken container en fil ligger i
- Container-størrelse eller -nummer

Lars ser bare sitt eget virtuelle fil-tre. Resten er administrativt arbeid appen håndterer.

## 3 · Tre redigeringsmodeller — og hvorfor vi valgte hybrid

### 3.1 Modell 1 — Download/Upload-loop (universell baseline)

```
Lars klikker "Rediger kontrakt.docx"
  → Ko|Do dekrypterer → trigger nedlasting → fila lander i ~/Downloads/
  → Permanent banner i Ko|Do: "✏️ Redigerer kontrakt.docx — [Last opp endret versjon]"
  → Lars åpner i Word, redigerer, Cmd+S → Word skriver til ~/Downloads/kontrakt.docx
  → Lars klikker "Last opp endret versjon" → file-picker → velger fila
  → Ko|Do krypterer → erstatter i container → re-uploader til Drive
```

**Pros:** Fungerer overalt (alle browsere, alle plattformer, iOS, iPad).
**Cons:** To manuelle klikk per redigering. Klartekst-kopi blir liggende i Downloads.

### 3.2 Modell 2 — File System Access API (smart redigering for støttede plattformer)

```
Førstegangs-oppsett (én gang per browser-profil):
  Lars klikker "Aktiver smart redigering" → showDirectoryPicker() → velger ~/KoDoEditing
  → Handle lagres i IndexedDB

Hver redigering:
  Lars klikker "Rediger kontrakt.docx"
  → Ko|Do dekrypterer → skriver til ~/KoDoEditing/kontrakt.docx via handle
  → Polling-loop: every 3 sec sjekk lastModified
  → Lars åpner i Word, redigerer, Cmd+S
  → Ko|Do oppdager endring innen 3 sek
  → Auto-re-encrypt → upload container → toast: "Synkronisert ✓"
```

**Plattform-støtte (per februar 2026):**

| Plattform | Browser | Modell 2? |
|---|---|---|
| Mac/Win/Linux | Chrome, Edge | ✅ |
| Mac | Safari | ❌ |
| Mac/Win/Linux | Firefox | ❌ |
| iPad | Alle (Safari-engine pålagt av Apple) | ❌ |
| iPhone | Alle (Safari-engine pålagt av Apple) | ❌ |
| Android | Chrome | ✅ |

**Cons:** Polling er hacky (FileSystemObserver-API kommer 2027). Permissions kan utløpe.

### 3.3 Modell 3 — Native app (AVVIST)

Electron/Tauri som monterer virtuell disk. Bryter "web-only, null install"-fortellingen. Ut.

### 3.4 Modell 4 — «Bruk Cryptomator» (AVVIST)

Å gi opp redigerings-ambisjon helt. Ko|Do mister konkurransefortrinn. Ut.

### 3.5 VALGT: Hybrid (Modell 1 universell + Modell 2 som power-mode + Klasse A native editor)

**Faseringsplan:**
- **v4.5.0:** Modell 1 universelt + Klasse A-editor (notater, markdown i Ko\|Do)
- **v4.5.5:** Modell 2 som opt-in "Smart redigering" på Chrome/Edge desktop + Android

**Hvorfor faseringen:**
1. Validerer produkt-fit først — Modell 1 + Klasse A er kanskje nok for de fleste
2. Modell 1 er det Lars uansett må forstå (mental modell)
3. FileSystemObserver-API modnes til 2027 → Modell 2 blir bedre med tid
4. Mindre v4.5.0-scope → raskere lansering

## 4 · Klasse A vs Klasse B-filer (kritisk distinksjon)

Lars sine filer deles i to klasser med ulik redigerings-flyt:

### 4.1 Klasse A — Innfødte Ko\|Do-filer (redigeres ALDRI ut av appen)

- 📝 **Notater** (markdown med rich-text-rendering)
- 📋 **Strukturerte dokumenter** (kontrakt-templates, sjekklister)
- 📊 **Enkle tabeller** (Ko\|Do sin mini-spreadsheet — ikke Excel-komplekst)

**Editor:** Rich-text inne i Ko\|Do, basert på **TipTap eller Lexical**.

**Features (v4.5.0 må-ha):**
- Markdown med rendering (fet, kursiv, lister, lenker, sjekk-bokser)
- Bilder embedded (drar-og-slipper inn, krypteres samme container)
- Enkle tabeller
- Søk i innhold per dokument (ikke cross-doc før v6)
- Versjons-historikk per auto-save (siste 10 versjoner)

**Estimert arbeid for Klasse A-editor alene:** 4-6 dager.

**Klartekst lever kun i RAM i Ko\|Do, aldri på disk.** Dette er sikrest-mulig på web.

### 4.2 Klasse B — Eksterne filer

- 📄 PDF
- 📝 Word (.docx)
- 📊 Excel (.xlsx)
- 🖼 Bilder (vises inline)
- 📦 Alt annet binært

**Lese-flyt:** Inline preview i Ko\|Do der mulig (PDF, bilder, text/csv/md — gjenbruker `PackagePreview`-komponenten fra v4.0).

**Rediger-flyt:** Modell 1 (download/upload) eller Modell 2 (FSAccess) avhengig av plattform.

### 4.3 Hvorfor splitten er viktig

Hvis Klasse A-editoren er god nok, kan Lars holde seg INNE i Ko\|Do for det meste av daglig arbeid. Word/Excel-flyt-en blir exception-path, ikke happy path. Det forsterker UX og reduserer eksponering av klartekst til disk.

## 5 · Plattform-strategi (eksplisitt segmentering)

### 5.1 iPad — bevisst posisjon, ikke unnskyldning

Apple krever at alle nettlesere på iPad bruker WebKit (Safari-motoren). Chrome, Firefox, Edge på iPad er bare Safari med annet UI. Modell 2 fungerer ikke.

**Posisjonering (Mike sin beslutning, 2026-02):**

> «iPad er fin for å lese pakker og bruke Klasse A-editor (notater). For full redigering av eksterne filer: bruk Mac, PC eller Android-tablet (f.eks. Samsung Galaxy Tab S11 Ultra).»

**Hvorfor dette er smart posisjonering:**
- Ærlig om hva Apple kontrollerer
- Lars får et **alternativ**, ikke en feilmelding
- Posisjonerer Ko\|Do som faglig integritets-driver, ikke "universal-app som unnskylder seg"
- 1Password/Tresorit kan ikke si dette så tydelig — de prøver å være alt for alle

### 5.2 Plattform-matrise

| Plattform | Klasse A-editor | Modell 1 (D/U) | Modell 2 (FSAccess) |
|---|---|---|---|
| Mac Chrome/Edge | ✅ | ✅ | ✅ (smart) |
| Mac Safari/Firefox | ✅ | ✅ | ❌ |
| Win/Linux Chrome/Edge | ✅ | ✅ | ✅ (smart) |
| Win Firefox | ✅ | ✅ | ❌ |
| Android Chrome | ✅ | ✅ | ✅ (smart) |
| iPad (alle browsere) | ✅ | ✅ | ❌ ikke mulig |
| iPhone (alle browsere) | ✅ | ✅ (download) | ❌ ikke mulig |

**Klasse A-editor fungerer overalt.** Det er den universelle baselinen.

## 6 · Streaming-prinsippet (terminologi-klargjøring)

**Streaming i Ko\|Do = chunked decrypt i nettleser. Aldri klartekst-streaming fra server.**

```
Drive    →  sender 4 MB ciphertext  →  Lars sin browser
                                          ↓
                                       Web Crypto API
                                       dekrypterer i RAM
                                          ↓
                                       Lokal Blob
                                          ↓
                                       PDF-viewer / editor
                                          ↓
                                       Lukket → Blob slettes
```

**Drive ser aldri klartekst.** Ko\|Do-server er ikke involvert i flyt-en (browser ↔ Drive direkte via OAuth).

For store filer (>50 MB) brukes Web Streams API for å dekryptere chunk-vis uten å laste alt i RAM. Dette er ikke kritisk for v4.5.0 men listet i åpne spørsmål.

## 7 · Søk-strategi

**Beslutning (Mike + agent, 2026-02):** Filnavn + tags. Innholds-søk venter til v6.

### Hvorfor

- Filnavn-søk er enkelt og billig (filnavn er allerede i `catalog.enc`)
- Tags er familiær mental modell fra G.Keep, Apple Notes, Mac Finder
- Innholds-søk krever full-text-index → komplekst, kostbart å vedlikeholde
- Lars sier: «luksus i versjon 6.0»

### UI

- Cmd+K åpner søk-bar (samme mønster som passord-vault i hurtig-laget)
- Søker på: filnavn (substring + fuzzy) + tags (eksakt match)
- Tags styres manuelt av Lars (klikk på fil → "Legg til tag")
- Auto-suggest av eksisterende tags fra katalogen

## 8 · Master-passord-strategi (ÅPEN — diskuteres før implementasjon)

To alternativer:

### Alt A — Samme master-passord for begge lag
- ✅ Lars husker bare ett passord
- ✅ Enklere UX, samme login-flyt
- ❌ Brudd-radius: kompromittert master gir tilgang til ALT (passord + dokumenter)

### Alt B — Forskjellige master-passord per lag
- ✅ Brudd-radius: kompromittert hurtig-lag-passord gir ikke dokument-tilgang
- ❌ Lars må huske to passord
- ❌ Komplisert UX (hvilket passord brukes nå?)

### Hybridforslag (foreløpig agent-anbefaling)
- **Samme master-passord** (Alt A) for enkelhets skyld
- **Men:** Egen salt + egen KDF-iter for dokument-laget → kryptografisk uavhengighet
- Det betyr: brute-force på Upstash-blob hjelper ikke for Drive-containere selv om angriper har begge

**Beslutning utsatt til implementasjons-tidspunkt.**

## 9 · Åpne arkitektur-spørsmål

Disse må besvares før eller under implementasjon:

### 9.1 Container-allokerings-strategi (KRITISK)

Hvordan deles filer mellom containere?

| Variant | Pro | Con |
|---|---|---|
| **a) Mappe-basert** (én container per mappe-gruppe) | Intuitiv for Lars | Ujevn størrelse |
| **b) Bin-packing FFD** | Optimal plass | Relaterte filer havner ikke sammen |
| **c) Tidsbasert** | Nylig brukt = én container | Endring av gammel fil = re-organisering |
| **d) Auto-clustering** | Bruks-mønster styrer | Kompleks, "magisk" |

**Agent-anbefaling:** Start med **mappe-basert (a)**, evt. bin-packing innen mappe ved overskridelse. Senere kan vi tilby "Optimaliser disken"-knapp som bytter til auto-clustering.

### 9.2 Container-størrelse — soft vs hard limit

**Forslag:** Soft target 10 MB, hard maks 15 MB. Splitt automatisk når over.

### 9.3 «Fat container»-problem

Hva med en 50 MB-fil i et 10 MB-system?

**Forslag:** Splittes IKKE på tvers (komplekst). Den ene fila får sin egen "fat container" og merkes i katalogen. Maks fil-størrelse bestemmes av Drive-grense (5 TB per fil — i praksis aldri grense).

### 9.4 Fragmenterings-strategi

Etter mye sletting kan en container være halvfull. Strategier:

| Variant | Hvordan |
|---|---|
| **Manuell vacuum** | Lars trykker "Rydd opp" → app slår sammen små containere |
| **Auto-vacuum @ 40% fragmentering** | Triggrer som bakgrunns-jobb |
| **Ignorer** | Drive-lagring koster Lars ingenting → bare et UX-problem |

**Agent-anbefaling:** Auto-vacuum @ 40% fragmentering, kjøres i bakgrunnen mens Lars jobber. Ikke krav om brukerhandling.

### 9.5 Katalog-recovery

`catalog.enc` er single-point-of-failure. Beskyttelse:

1. **Drive-versjons-historikk** — bevarer 30 dager automatisk
2. **Selv-beskrivende containere** — hver container har header med liste over egne filer → katalog kan rebygges
3. **Backup-katalog i Upstash** — liten kopi (~50 KB) lagres også i hurtig-laget

**Agent-anbefaling:** Implementer alle tre. Krypto-applikasjoner må overleve katastrofer.

### 9.6 Cross-device sync-konflikter

Lars endrer fila på Mac, glemmer å lukke, åpner samme fila på iPad.

**Forslag:**
- Hver container har `etag` (checksum av ciphertext)
- Før upload: sjekk Drive sin nåværende etag mot lokalt cached etag
- Hvis ulik → konflikt-dialog: "Mac-versjon (12:34) eller iPad-versjon (12:36)?"
- Tap-toleranse: behold begge versjoner midlertidig i en `conflicts/`-mappe

### 9.7 OAuth-scope

| Scope | Hva Lars ser i Drive |
|---|---|
| `drive.appdata` | Ingenting (skjult mappe, kun Ko\|Do-app kan se) |
| `drive.file` | Mappe `Ko\|Do Vault/` med .enc-filer (Lars ser dem) |

**Agent-anbefaling:** `drive.file` med mappe `Ko\|Do Vault/`. Lars kan se at noe ligger der → forstår BYO-konseptet → kan flytte filer ut hvis han vil bytte sky.

### 9.8 Krypto-format

| Variant | Hva |
|---|---|
| **Gjenbruk `.kodoenc` v4.0 (format-byte 0x01)** | Containere er bare flere `.kodoenc`-filer |
| **Nytt format-byte 0x10 (vault-container)** | Eget format for v4.5 med ekstra metadata |

**Agent-anbefaling:** Gjenbruk `.kodoenc` med ny **header-flag-bit** (D-027 §reservert 3 bytes). Hver container er en gyldig `.kodoenc` Lars kunne pakket ut manuelt om Ko\|Do forsvant.

### 9.9 Editing-flow for Klasse B — temp-fil-lifetime

Modell 1: Klartekst i Downloads til Lars sletter manuelt.
Modell 2: Klartekst i `~/KoDoEditing/` så lenge Lars åpner / til auto-cleanup.

**Forslag for Modell 2:** Etter Lars markerer "Ferdig redigert" → auto-slett lokal kopi (best effort).

### 9.10 Mobil-RAM-grense

Kan en iPad realistisk holde 200 containere á 10 MB i RAM (= 2 GB)?

**Svar:** Nei. iPad har typisk 4-8 GB RAM, men Safari får ~1 GB.

**Mitigering:** Cache kun N siste containere i RAM, evict eldste når over grense.

### 9.11 Notes vs filer

Er notater (Klasse A) egen blob-type eller bare filer i en spesiell container?

**Agent-anbefaling:** Egen "notes container" (en eller flere) som behandles likt som andre containere arkitektur-messig, men har sin egen UI (rich-text editor i stedet for preview).

### 9.12 Thumbnail-strategi

Forsiktig grad av thumbnail-generering for bilder/PDF:

| Variant | Konsekvens |
|---|---|
| **Embed thumbnail i catalog.enc** | Katalog blir større, men ett kall |
| **Egen `thumbnails.enc`-container** | Last lazy ved behov |
| **Ingen thumbnails** | Generic icons, raskeste implementasjon |

**Agent-anbefaling:** Start uten thumbnails (v4.5.0). Legg til som v4.5.x-enhancement når brukere ber om det.

## 11 · Trade-offs vi aksepterer

Disse er bevisste valg, ikke kompromiss:

1. **iPad har redusert funksjonalitet for Klasse B-redigering.** Vi fortller Lars: "bruk Mac, PC eller Android-tablet for full redigering." Ikke en feilmelding — et alternativ.

2. **Klartekst-Klasse-B-filer lever på disk under redigering.** Vi prøver ikke å skjule det. Akkurat som å ta papir ut av en safe — Lars må selv legge det tilbake.

3. **Innholds-søk støttes ikke i v4.5.** Lars må bruke filnavn + tags. v6 kan legge til innholds-søk hvis det er etterspurt.

4. **Single-cloud (Drive) i v4.5.0.** Multi-cloud (Dropbox, OneDrive, iCloud, lokal USB) blir egen versjons-bump (v4.6+). BYO-prinsippet er bevart — Lars eier dataene og kan flytte dem manuelt. Landingssida (kodovault.no) lover Drive/Dropbox/USB — vi må levere multi-cloud før v5.0-launch.

5. **Modell 2 ikke i v4.5.0.** Vi sender Modell 1 først, validerer behov, så bygger Modell 2 hvis brukere ber om det.

## 12 · Avviste alternativer (med begrunnelse)

| Avvist | Hvorfor |
|---|---|
| **Native Electron/Tauri app** | Bryter "web-only, null install"-fortellingen. Mike sin LinkedIn-pitch dør. |
| **Service worker for offline `/unpack`** | Tilstedeværelse-paradokset — krever at Anna besøker sida mens online. Lottokupong. (Diskutert under v4.0, gjelder også her.) |
| **Standalone HTML-decryptor distribuert med pakka** | Lars roter den bort. Mismatch mot nye versjoner. |
| **Knapp i appen som genererer offline-kopi** | Forutsetter at Lars er online akkurat da. |
| **Universal Drive-monteringsverktøy** | Krever native app per OS. |
| **Cryptomator-alternativ med samme native model** | Konkurrerer direkte mot etablert spiller. Vi har ikke distribusjons-fordel. |
| **Byte-nivå-splitting (RAID/par2-style)** av containere | Alt-eller-ingenting-sikkerhet ikke verdt UX-kostnaden (samme D-028-argument fra v4.0). |
| **TOTP/2FA i pakker** | Egen feature i hurtig-laget (v4.2). Uavhengig av v4.5. |

## 13 · Sammenheng v4.0 ↔ v4.5

`.kodoenc`-formatet fra v4.0 (D-027) gjenbrukes som container-format. Hver container er en gyldig `.kodoenc` Lars kunne pakket ut manuelt med UnpackModule om Ko\|Do forsvant. **Det er Lars sin nød-utgang — han eier dataene 100%.**

Klasse A-editor er ny i v4.5 og uavhengig av v4.0.
