# Ko|Do · Vault — v4.1 SPEC (ID-blob — egen blob for ID-dokumenter)

**Versjon:** v6 utkast (uignert)
**Dato:** 2026-02
**Status:** Under sign-off — implementasjon starter når Mike godkjenner
**Estimert iter-omfang:** 4-5 iterasjoner

## 0. Hvorfor denne spec'en finnes

v4.1 er andre store videreutvikling etter v4.0. Vi splitter cards-arbeidet i to (per D-013):
- **v3.0** ✅ I PROD — Bankkort (betalingskort med PIN/CVV)
- **v4.1** ⏳ ID-blob — Pass, førerkort, helsekort, forsikring (ikke-betalings-dokumenter)

**Hva v4.1 IKKE er:**
- ❌ Generell dokument-lagring (det er v4.5)
- ❌ Bankkort (det er v3.0, allerede i prod)
- ❌ Et nytt design-paradigme — vi gjenbruker v3.0-mønstre tett

**Mike sin formulering 2026-02:** «Tanken er at et forsikringsbevis kommer på mail eller pdf fra forsikringsselskapet, og ikke annet. Det har ikke noe med dokumenter å gjøre.»

Det er definisjonen som styrer hele spec'en: **vedlegg = bevis på ID/medlemskap/dekning** — ikke vanlige dokumenter.

## 3. Strukturerte typer (i blob `vault:default:ids`)

| Nasjon | enum | ✓ | Norge / Sverige / Annet |
| Passnummer | string | ✓ | C12345678 |
| Utstedt av | string | – | Politiet, Oslo |
| Utstedt dato | date | – | 2024-03-15 |
| Utløpsdato | date | ✓ | 2034-03-14 |

### 3.3 ID-kort 🆔

For bank-ID-er (BankID-kort), Nasjonalt ID-kort, ansatte-ID-er, medlemskort til offisielle organer.

| Felt | Type | Påkrevd? | Eksempel |
|---|---|---|---|

| Vedlegg | image/PDF | – | Maks 1 MB |
| Notater | textarea | – | Fri tekst |

### 3.4 Helsekort / forsikring 🏥

For helse-relatert ID + alle typer forsikringer (reise, bil, hjem, livsforsikring).

| Felt | Type | Påkrevd? | Eksempel |
|---|---|---|---|
| Type | enum | ✓ | Helsetrygdekort / Reiseforsikring / Bilforsikring / Annet |
| Selskap | string | ✓ | NAV / If / Gjensidige |
| Polise-/medlemsnr | string | ✓ | 12345-67-890 |
| Kontakt-tlf | string | – | +47 21 49 24 00 |
| Kontakt-e-post | string | – | skade@if.no |
| Gyldig fra | date | – | 2025-01-01 |
| Gyldig til | date | ✓ | 2025-12-31 |
| Vedlegg | image/PDF | – | Maks 1 MB |
| Notater | textarea | – | Fri tekst (dekningsdetaljer osv.) |

### 3.5 Hva som IKKE er en ID-type i v4.1

- ❌ **Betalingskort** (Visa, Mastercard) — det er v3.0 Bankkort
- ❌ **Lojalitets-kort uten foto/nummer** (Coop-medlemskap, kaffe-kort) — passord-vault holder for de
- ❌ **Reseptpliktig medisin / journal** — det er sensitivt på et annet nivå, vurderes senere
- ❌ **Skolevitnemål, sertifikater, attester** — det er v4.5 (vanlige dokumenter)

Grunnregel: **Hvis det har en utløpsdato eller offisielt nummer, og du må kunne vise det for å bevise noe — er det v4.1.**

## 4. Vedleggs-spesifikasjon

### 4.1 Tillatte formater

| Format | MIME-type | Komprimering |
|---|---|---|
| JPEG | `image/jpeg` | Re-encode til 80% kvalitet, maks 1600px lengste side |
| PNG | `image/png` | Re-encode til JPEG hvis < ~70% kvalitets-tap, ellers behold PNG |
| WebP | `image/webp` | Konverter til JPEG (bredere kompatibilitet) |
| HEIC | `image/heic` | Konverter til JPEG (iPhone-kompatibilitet) |
| PDF | `application/pdf` | Ingen komprimering — bruker fil som den er, valider størrelse |

### 4.2 Størrelses-grenser

- **Per vedlegg:** Hard maks **1 MB** etter komprimering (Mike-beslutning 2026-02)
- **Per ID:** Maks 1 vedlegg (foto ELLER PDF, ikke begge)
- **Total ID-blob:** Soft target ~25 MB. Upstash record-grense er 100 MB; vi tar 75% margin.

### 4.3 Maks antall ID-er

- Hvis hver ID gjennomsnittlig har ~800 KB vedlegg + strukturerte data: realistisk ~30 ID-er
- Det dekker ekstreme power-users (Lars med pass + førerkort + 5 forsikringer + 10 medlemskort)
- Når Lars nærmer seg 20 MB: vis advarsel «Du nærmer deg max — vurder å slette gamle»

### 4.4 Kilde

Per Mike-beslutning 2026-02:
- **Fil-picker** (primær, alle plattformer): `<input type="file" accept="image/*,application/pdf">`
- **Kamera-flow** (når enheten har kamera): gjenbruker v3.0 CardCamera + CardCropper med per-type crop-guides (se §4.5)
- **Drag-and-drop** (desktop): drop-zone i Edit-modal — i scope for v4.1.0

### 4.5 Kamera-flow — per-type crop-guides

Gjenbruker v3.0-flowen fra Bankkort. Forskjellen: per-ID-type-aspect-guide.

| ID-type | Aspect-ratio guide | Format |
|---|---|---|
| 🛂 Pass | ~1.4:1 portrait | Passport-page (åpen oppslag) |
| 🚗 Førerkort | 1.586:1 landscape | ISO/IEC 7810 ID-1 (samme som kort) |
| 🆔 ID-kort | 1.586:1 landscape | ISO/IEC 7810 ID-1 |
| 🏥 Helse/forsikring | 1.586:1 landscape | ISO/IEC 7810 ID-1 |

**Implementasjon:**
- Lars velger ID-type FØRST (Pass / Førerkort / ID-kort / Helse)
- Når han trykker «Ta bilde», får han type-spesifikk guide
- Pass åpnes med vertikalt rektangel, andre med horisontalt rektangel
- Gjenbruker CardCamera-komponenten — utvides med `aspectMode: "id-1" | "passport"`-prop
- Cropper-stadiet bruker samme guide som kamera-stadiet

## 5. Teknisk arkitektur

### 5.1 Blob-modell

```
Upstash keys (per tenant):
  vault:default:meta       — KDF-salt + iter (allerede der fra v2.9)
  vault:default:passwords  — Passord-blob (allerede der)
  vault:default:cards      — Bankkort-blob (allerede der fra v3.0)
  vault:default:ids        ← NY i v4.1
```

### 5.2 Krypto-isolering

- **Samme master-passord** som passord + cards (samme KDF-derivert nøkkel)
- **Egen salt og IV** per blob (kryptografisk uavhengig)
- Hvis angriper får tilgang til ID-blob alene → må fortsatt brute-force master-passord via PBKDF2
- Per D-001 (zero-knowledge): klartekst forlater aldri Lars sin enhet

### 5.3 Lazy load

- ID-blob lastes **ikke** ved app-oppstart
- Lastes første gang Lars klikker 🆔-fanen i samme sesjon
- Caches i RAM resten av sesjonen (til auto-lås)
- På auto-lås: cache slettes umiddelbart

### 5.4 Selektiv backup (per D-021)

- Backup-eksport-flow må håndtere 3 blober: passwords, cards, ids
- Smart re-kryptering: hvis Lars endrer master-passord, re-krypteres alle 3 blober separat
- Bruker bruker eksisterende `BackupExportModal` med ekstra blob-type

### 5.5 Komprimerings-pipeline (vedlegg)

```
[Filvalgt] → [Sjekk størrelse] → [Sjekk type]
                                       ↓
                                  ├── PDF: behold som er, valider <= 1 MB
                                  └── Image: re-encode JPEG 80%, maks 1600px
                                              ↓
                                       valider <= 1 MB
                                              ↓
                            [Encrypt med blob-nøkkel] → [Lagre i blob]
```

### 5.6 Vannmerke-funksjon

Klient-side canvas-rendering:

```
1. Lars klikker «Eksporter med vannmerke» på en ID
2. Modal: «Hva er formålet med delingen?»
   Input-felt: f.eks. «Hertz Bil, 14.02.2026»
3. Ko|Do rendrer canvas:
   - Originalt foto i bakgrunnen
   - Diagonalt overlay-tekst: "KUN FOR HERTZ BIL – 14.02.2026 – Lars Larsen"
   - Semi-transparent (40-60%)
4. Canvas → toBlob() → trigger nedlasting som PNG
5. Original-foto i vault uendret
```

**Hvorfor vannmerke er viktig (D-001-kompatibel):**
- Vannmerket binder bildet til konkret mottaker + dato
- Hvis bildet lekker → spor til hvem som leakte
- Diskuranerer hot-deling av samme bilde til flere
- Lars beholder originalen privat i vault-en

PDF-vannmerke: **ikke i v4.1** (krever pdf-lib + re-rendering, ~2 dager arbeid). Hvis vedlegg er PDF → "Eksporter"-knapp eksporterer original-PDF uten vannmerke. Dokumenteres som begrensning.

## 7. Edge cases + sikkerhet

### 7.1 D-001 100%-sjekkliste for v4.1

- [x] Vedlegg krypteres før upload — klartekst forlater aldri Lars sin enhet
- [x] Master-passord kreves for å laste/dekryptere ID-blob
- [x] Auto-lås tømmer ID-blob fra RAM
- [x] Eksport med vannmerke skjer klient-side (canvas) — server ser aldri klartekst-bilde
- [x] Backup inkluderer ID-blob i kryptert form
- [x] PDF-vedlegg behandles identisk med foto-vedlegg krypto-messig

### 7.2 Ekstra sikkerhets-konsekvenser av PDF-vedlegg

- PDF kan inneholde **embedded JavaScript** (sjelden i offisielle dokumenter, men mulig)
- **Tiltak:** Inline PDF-viewer (PDF.js) i sandbox-modus, JS disabled
- PDF kan inneholde **embedded metadata** (forfatter, software, GPS-koordinater)
- **Tiltak:** Vi viser ikke metadata til Lars, men de er fortsatt i fila — informeres i FAQ

### 7.3 Vedlegg-erstatning

- Lars erstatter vedlegg → gammelt bytes overskrives i blob → nye uploades
- Gammelt vedlegg er **borte fra Upstash** etter neste sync (ingen versjons-historikk på server)
- Lars sin egen Drive-versjons-historikk gjelder kun for v4.5, ikke v4.1

### 7.4 Sletting av ID

- Bekreftelse-modal: «Slett pass Norge?» (Rose-farge per D-031)
- Vedlegg fjernes fra blob
- Audit-event opprettes (hvis events_logged-feature er på) → loggføring av sletting

### 7.5 Migrering fra v3.0 Bankkort

**Beslutning:** Bankkort i v3.0 fortsetter som er. Ingen migrering til v4.1.

Hvis Lars har lagret kort som «ID» i v3.0 av misforståelse — det er hans valg. Vi gjør ikke automatisk flytting.

## 9. Avklarte beslutninger (2026-02)

### 9.1 Hva er v4.1, hva er v4.5
- **v4.1:** Strukturerte ID-dokumenter med felt + maks 1 vedlegg
- **v4.5:** Generell dokument-lagring (PDF/Word/bilder uten struktur, mange filer)
- Forsikrings-PDF (1 side) hører hjemme i v4.1
- 50 MB skatte-papir-arkiv hører hjemme i v4.5

### 9.2 Fire ID-typer (ikke fem)
Cards-typen fra ROADMAP-skissen droppes — det er v3.0. Vi har: Pass, Førerkort, ID-kort, Helse/forsikring.

### 9.3 Maks 1 vedlegg per ID
Forenkler datamodell og kvote-håndtering. Hvis Lars vil ha forside + bakside av førerkort → bruker bare ett bilde med begge sider scannet sammen.

### 9.4 Søk via Cmd+K, ikke egen søke-flate
ID-er er sjeldne nok at dedikert søke-UI er overkill. Cmd+K skalerer.

### 9.5 Vannmerke-eksport som primær delings-mekanisme
Per Lars-fortellingen: dele bilde med leiebil-firma. Vannmerke binder bildet til mottaker + dato.

### 9.6 Kamera-flow MED per-type crop-guides
Gjenbruker v3.0 CardCamera + CardCropper. Utvides med `aspectMode`-prop: `"id-1"` for kort-format, `"passport"` for pass. Lars velger ID-type først, kameraet viser type-spesifikk guide.

### 9.6b Drag-and-drop i scope
Drop-zone i Edit-modal som alternativ til fil-picker og kamera. Desktop-vennlig.

### 9.6c Tre kilder for vedlegg
1. Fil-picker (primær, alle plattformer)
2. Kamera-flow (med per-type guides)
3. Drag-and-drop (desktop)

### 9.7 Egen 🆔-fane (ikke underseksjon)
ID-er er konseptuelt forskjellig fra Bankkort. Egen fane gir tydeligere mental modell.

### 9.8 Vannmerke kun på bilder
PDF-vannmerking utsettes (krever pdf-lib). Eksport av PDF gir original-fil.
