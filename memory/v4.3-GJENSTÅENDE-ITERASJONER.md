### Iter 13.5 — `/api/billing/checkout-info` (GET)

**Mål:** Endepunkt som betalingsvegg og upgrade-side kan lese.

**Leveranse:**
- `GET /api/billing/checkout-info`
- Leser subdomain fra host-header (D-046)
- Returnerer ved suksess:
  ```json
  {
    "ok": true,
    "status": "trial" | "locked" | "active",
    "trialEndsAt": "2026-07-06T...",
    "daysRemaining": 18,
    "hasStripeCustomer": true,
    "plan": "trial" | "monthly" | "yearly"
  }
  ```
- Ingen auth — kun subdomain-basert

**Feil-responser (eksplisitt spec):**
| Tilfelle | Status | Body |
|---|---|---|
| Tenant ikke funnet | `404` | `{ ok: false, error: "tenant_not_found" }` |
| Host er admin/root/www/api/localhost | `400` | `{ ok: false, error: "invalid_host" }` |
| Tenant-status er ikke `trial` eller `locked` | `400` | `{ ok: false, error: "invalid_status" }` |
| Missing host-header | `400` | `{ ok: false, error: "missing_host" }` |

- Backend (D-076 — cache-sync write-block):
  - `lib/server/tenant-status-cache.ts` — `subdomainFromHost()`, `getTenantStatus()`, `assertTenantNotLocked()`, `checkWriteBlock()`. Tenant-poden cacher `{status, lockedAt}` i lokal Upstash med TTL 5 min, fetcher fra admin via RPC ved miss.
  - `app/api/internal/tenant-status/route.ts` — admin RPC-endepunkt (Bearer-beskyttet via `INTERNAL_RPC_SECRET`)
  - Write-block anvendt på: `PUT/DELETE /api/vault`, `PUT/DELETE /api/cards`, `PUT/DELETE /api/ids`

**Nye env-vars:**

### Original spec (referanse — implementert per 2026-06-13)

**Mål:** Automatisk håndtering av lifecycle-overganger via daglig cron.

**Forutsetninger:** Iter 15 (Telegram) og Iter 16 (Resend e-post) er allerede levert som del av Iter 10. Bruk eksisterende sende-helpers.

**Tidslinje (D-075 · 2026-06-13 · KORRIGERT — ingen "cancelled"-mellomsteg):**

| Tidspunkt | Betingelse | Handling |
|---|---|---|
| `trialEndsAt - trialWarningDaysBefore` (default: dag 25) | `lifecycle: true` | E-post "X dager til prøveperioden utløper" + link til `/billing/upgrade` + Telegram til Mike |
| `trialEndsAt` (default: dag 30) | Ikke betalt | `status: "locked"` + `lockedAt = nå` + E-post "Vaulten er låst, betal for å fortsette" + Telegram |
| `lockedAt + (lockToDeleteDays - deleteWarningDaysBefore)` (default: dag 51) | — | E-post "Sletter om X dager" (siste varsel) |
| `lockedAt + lockToDeleteDays` (default: dag 58) | — | `deleteTenant()` kaskade + E-post "Vaulten er slettet" + Telegram |

**Eksplisitt fjernet fra spec (D-075):**
- ~~Dag 37 (ren purring)~~ — ingen statusendring, droppet
- ~~Dag 44 (`status: "locked"` hard)~~ — overflødig siden status allerede er "locked" fra dag 30; vi hopper rett til "sletter om X dager"-varsel

**Lifecycle-tall lest fra config (`default.json`):**
```json
"lifecycle": {
  "trialDays": 30,
  "trialWarningDaysBefore": 5,
  "lockToDeleteDays": 28,
  "deleteWarningDaysBefore": 7
}
```

Brukes via `getLifecycle(subdomain)` fra `lib/platform/client-config-store.ts`.

> Alle destruktive cron-handlinger MÅ kalle `canAutoLock()` / `canAutoDelete()` (D-069). Håndheves av `yarn lint:d069`.
